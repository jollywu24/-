# 任务清单

本文件记录架构整理过程中发现的后续工作。以下项目均为建议，不在本次轻量级整理中执行。

## 已完成的第一阶段整理

- 已将红眼赌注、赌鬼、卡包和赌鬼展示说明集中到 `web/game-content.js`。
- 已将统一运行时状态初始化与重置提取到 `web/runtime-state.js`。
- 已将纯回合判定提取到 `web/round-rules.js`。
- 已将通用动画工具和 DOM 查询分别提取到 `web/animations.js`、`web/game-dom.js`。
- 已增加架构模块测试和固定 seed 浏览器基础流程测试。

## 最值得优先重构的 3 个点

1. 继续拆分 `web/app.js` 中的商店、红眼生命周期和领域动画编排。
2. 将赌鬼实际规则实现进一步接入集中定义，减少 `game-content.js` 与 `logic-pure.js` 的双处维护。
3. 扩展固定 seed 浏览器测试，覆盖失败、通关、商店与红眼赌注流程。

## P0：优先处理

### 修正 GitHub Pages 美术资源发布

**现状**

- 工作流只上传 `web/`。
- 页面通过 `../art/` 引用商店图和红眼 VFX。

**风险**

- 本地运行正常，但 GitHub Pages 可能缺少图片资源。

**建议后续重构**

- 选择统一资源发布策略：扩大 Pages artifact 范围，或将运行时资源放入可发布目录。
- 增加部署后资源可访问性检查。

### 增加浏览器流程回归测试

**现状**

- 已有固定 seed 浏览器测试覆盖初始化、选牌、换牌、摊牌和重载。
- 商店、红眼弹窗、Tips、通关与失败流程仍依赖人工检查。

**建议后续重构**

- 扩展现有固定 seed DOM 级测试。
- 优先新增：爆牌、通关、商店购买、红眼赌注。

### 明确当前规则文档的唯一来源

**现状**

- 部分旧文档与当前代码存在阈值或接入状态偏差。

**建议后续重构**

- 以 `logic-pure.js` 测试作为单手规则数值事实来源，以 `round-rules.js` 测试保护纯回合规则。
- 更新旧文档中的历史阈值、商店状态和图片资源约束。
- 明确 `logic-pure.js` 是单手规则来源，`round-rules.js` 是纯回合规则来源，`app.js` 负责流程提交。

## P1：高价值架构改进

### 拆分 `web/app.js` 的职责

**现状**

`app.js` 仍同时负责：

- 输入事件。
- 商店。
- 红眼生命周期提交。
- 结算流程。
- 领域动画调度。

**风险**

- 修改局部流程容易影响无关区域。
- 难以为完整回合编写测试。

**建议后续重构**

- 继续提取商店、红眼生命周期和领域动画编排。
- 保留现有行为和数据结构，避免一次性重写。

### 统一玩家术语与内部术语

**现状**

规则核心仍大量使用 `pressure`、`module`、`redline`、`ignitionSequence` 等历史命名。

**风险**

- 新开发者容易误判当前产品概念。
- 文档与代码检索成本高。

**建议后续重构**

- 在稳定测试保护下逐步统一内部命名。
- 不要只做全局替换，应先明确数据契约。

### 为动画建立统一调度边界

**现状**

- 动画依赖多个 `wait`、定时器、class 和动态克隆节点。
- 资源动画与 CSS 动画分别触发。

**风险**

- 动画中断、快速重开或阶段切换时可能残留节点/class。
- 动画时长调整容易破坏流程时序。

**建议后续重构**

- 建立可取消的动画调度器。
- 统一清理临时 DOM、class 和定时器。
- 保留资源加载失败时的降级路径。

## P2：后续维护

### 集中赌鬼定义

**现状**

- 赌鬼商品数据与显示说明已集中，但实际规则仍位于 `logic-pure.js`。

**建议后续重构**

- 保持 `game-content.js` 为规则参数、商品字段和展示字段的单一注册表。
- 后续评估将规则处理器也注册到稳定接口，减少 `logic-pure.js` 中的 ID 分支。

### 明确卡包实现状态

**现状**

- 商店有卡包报价和购买入口。
- 降压道具有实际效果。
- 其他卡包当前主要消耗赌资和展示购买反馈，没有完整开包内容流程。

**建议后续重构**

- 明确卡包是占位演示还是下一阶段功能。
- 在实现前补充规则和测试。

### 处理 Python CLI 原型定位

**现状**

- `src/jokermvp/` 与 Web 版本规则不同，且没有共享代码。

**建议后续重构**

- 明确保留目的。
- 若仅作为历史原型，补充醒目标记或迁入归档区域。

### 建立资源清单和压缩流程

**现状**

- `art/` 包含高分辨率 PNG/JPG。
- 当前没有尺寸预算、压缩或预加载策略。

**建议后续重构**

- 记录资源用途、尺寸、体积和触发点。
- 对运行时资源评估 WebP、AVIF 或序列帧方案。
