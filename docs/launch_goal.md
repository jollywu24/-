# 上线 MVP 目标

本文档把“把 MVP 做到可公开上线，并服务 Steam 1000 个愿望单”拆成可执行目标。它不是当前玩法规则来源；规则仍以 `docs/component_rules.md` 为准。

## 北极星目标

公开试玩版要让陌生玩家在 5 分钟内理解并感受到：

> 我知道再摊一手可能爆牌，但我还是想贪。

Steam 愿望单目标为首轮验证目标：

- 短期目标：公开商店页后积累 `1000` 个愿望单。
- 判断方式：商店页转化材料是否能讲清“黑暗扑克 + 红眼上头 + 赌鬼下注”的差异化。
- 不以长线内容量作为第一优先级，不用局外成长、剧情、地图或装备系统掩盖核心循环。

## 上线定义

达到以下条件，才算“可以公开给玩家试玩”：

| 维度 | 上线标准 | 当前执行重点 |
| --- | --- | --- |
| 核心闭环 | 从发牌、选牌、换牌、摊牌、红眼、失败、通关、收益、商店到下一轮可连续运行 | 保持 `web/tests/` 覆盖并补手动 QA |
| 规则可信 | 上头值、红眼赌注、暗涌、欠命、跨轮代价和收益都可追溯到文档与测试 | 不经确认不改玩法规则 |
| 线上可玩 | GitHub Pages 或静态部署能加载 `web/`、`art/`、`public/assets/` | Pages artifact 必须包含运行时资源 |
| UI 稳定 | 固定 `1600 x 900` 画布无滚动，手机横屏可缩放，主 UI 不错位 | 不改五大常驻区域布局 |
| 失败可理解 | 爆牌、庄家通吃、通关收益、商店阶段都有清楚反馈 | 不新增最终结算页 |
| 转化材料 | 有 Steam 页面文案、截图清单、短视频镜头清单和美术资产缺口清单 | 先产出素材需求，不急着扩系统 |
| 反馈闭环 | 页面或试玩说明提供反馈入口，并能定位版本号或构建 SHA | Pages 注入 `window.__BUILD_SHA__` |

## 发布闸门

上线 MVP 分成三道闸门，必须按顺序推进。没有通过前一道，不进入下一道。

### Gate A：公开试玩闸门

目标：GitHub Pages 或等价静态入口可以交给玩家打开。

必须满足：

- 本地和线上入口都能打开 `/web/`，没有白屏或主资源缺失。
- 核心闭环可以连续运行到第二轮：发牌、选牌、换牌、摊牌、红眼、失败、重开、通关结算、商店、下一轮。
- 自动测试通过，浏览器流程测试或手动 QA 至少覆盖一次完整通关和一次失败。
- 规则文档与实现一致；任何规则争议必须先确认，不带着冲突上线。
- 反馈入口可用，且能让玩家提交版本号、是否理解玩法、是否想重开、是否愿望单。

### Gate C：1000 愿望单验证闸门

目标：用公开商店页和试玩版验证是否能拿到首批 `1000` 愿望单。

必须满足：

- 页面首屏能在 5 秒内讲清：黑暗扑克、上头风险、红眼赌注。
- 至少 5 张首发截图覆盖真实玩法状态，不重复展示同一种画面。
- 15-30 秒短视频能展示“选牌 -> 上头上涨 -> 红眼赌注 -> 暗涌 -> 结果”的情绪弧线。
- 反馈入口能追踪“是否理解玩法、是否主动红眼、是否想重开、是否愿望单”。
- 每次公开传播使用 UTM 或等价来源标记，后续能判断愿望单来自哪里。

当前判断：

- 文案草案、shot list、视频脚本和胶囊图需求已经有第一版。
- 真实首发截图、短视频/GIF、最终胶囊图和线上反馈入口尚未完成。

## P0：公开试玩前必须完成

1. **线上资源发布**
   - GitHub Pages artifact 必须包含 `web/`、`art/`、`public/`。
   - 线上入口应使用 `https://<用户名>.github.io/<仓库名>/web/`。
   - 打开后不得出现主要背景、卡背、红眼、暗涌图缺失。

2. **核心流程冒烟**
   - 新局初始化。
   - 选择 1-5 张牌并预览牌型、筹码、倍率。
   - 摊牌后分数、上头值、出牌区和补牌正常。
   - 换牌后弃牌、补牌、牌堆数量和剩余换牌次数正常。
   - 上头进入红眼后只解锁红眼入口，不自动弹窗。
   - 选择红眼赌注后只影响下一次摊牌。
   - 爆牌失败、庄家通吃失败和再来一局正常。
   - 通关结算、领取收益、商店、进入下一轮正常。

3. **测试基线**
   - `node --check web/app.js`
   - `node --check web/logic-pure.js`
   - `node --check web/round-rules.js`
   - `node --test web/tests/logic-pure.test.mjs web/tests/architecture-modules.test.mjs`
   - `node --test web/tests/deploy-assets.test.mjs`
   - `node --test web/tests/browser-flow.test.mjs`
   - `git diff --check`
   - `.github/workflows/test-web.yml` 在 GitHub Actions 中运行同一套基线；若当前环境无法启动无头浏览器，需要保留手动 QA 清单作为替代证据。

4. **规则文档一致**
   - 当前规则只看 `docs/component_rules.md` 和领域 README。
   - 规则、流程、资源路径或测试方式变化时，必须同步文档。
   - Boss 暂不进入上线 P0。

5. **首轮转化材料**
   - 一句话卖点。
   - Steam 短描述与长描述草案。
   - 5 张截图镜头清单。
   - 15-30 秒短视频或 GIF 镜头清单。
   - 胶囊图需求清单。
   - 玩家反馈入口方案。

## P1：服务 1000 愿望单的转化工作

### 定位原则

商店页第一屏只卖一个体验：

> 黑暗扑克肉鸽里，分数越贪，上头越危险；红眼下注可能救你，也可能把整局送掉。

所有文案和视觉素材都应服务这三个关键词：

- **扑克牌心智**：一眼知道自己在凑牌型、算筹码和倍率。
- **红眼风险**：一眼知道上头值在逼近爆牌。
- **赌鬼诱惑**：一眼知道有人在诱导你继续赌。

不要把第一批 Steam 文案写成世界观介绍。`深渊` 是气质，不是卖点本体。

### Steam 页面最小材料

| 材料 | 目标 |
| --- | --- |
| 一句话卖点 | 用一句话说明“扑克 + 上头值 + 红眼下注”的差异化 |
| 短描述 | 直接讲清玩法差异，不写世界观散文 |
| 长描述 | 依次说明核心循环、红眼赌注、赌鬼、风险失败、商店成长 |
| 截图 | 至少 5 张，分别覆盖主界面、结算、红眼赌注、暗涌/爆牌、通关/商店 |
| 短视频 | 15-30 秒，展示一次完整“贪一手”的情绪弧线 |
| 胶囊图 | 先做主视觉方向和尺寸清单，再进入美术制作 |

### Steam 商店文案草案

#### 一句话卖点

```text
在黑暗扑克赌桌上凑牌、爆分、压上头值，然后决定要不要用红眼赌注再贪一手。
```

#### 短描述草案

```text
《深渊赌局》是一款黑暗扑克肉鸽原型。你用标准扑克牌凑牌型赚分，用赌鬼效果推高倍率，同时管理不断上涨的上头值。进入红眼后，可以选择危险赌注翻暗涌牌：救你过线，或让整局爆牌。
```

#### 长描述草案

```text
每一局，你都从 8 张手牌中选择最多 5 张摊牌。对子、两对、顺子、同花、葫芦等标准牌型会提供筹码和倍率，你需要在有限摊牌次数内达到目标分数。

但每次得分都会让上头值上涨。上头值进入红眼后，你可以解锁一次红眼赌注：红眼翻倍、借鬼钱、偷过线、欠命……它们只影响下一次摊牌，却可能留下跨轮代价。

红眼赌注不会让你安全。它会翻开一张暗涌牌，用未知风险替代本手上头增长。翻到小牌，你可能极限过线；翻到大牌，你可能直接爆牌，分数够了也算输。

通关后领取收益，进入商店购买赌鬼或降压道具，再进入下一轮。目标分数继续上涨，上头值不会完全归零，赌桌会越来越热。

这不是一个靠剧情推进的原型。它只验证一件事：你明知道再贪一手可能会死，但你还是想点下摊牌。
```

### 截图 Shot List

截图必须来自真实可玩 UI，不使用额外营销排版盖住核心信息。

截图生产不在项目里维护自制截图脚本。优先使用现成工具：

- Codex Browser / Chrome / Edge 的截图能力。
- Playwright 等成熟工具链，如果后续项目正式引入浏览器自动化依赖。
- 当前 `web/tests/browser-flow.test.mjs` 只作为流程 QA 证据，不承担素材生产职责。

若需要批量截图，先确认工具选型和依赖边界，再进入实现；不要为了截图任务在仓库里重复造轮子。

| 序号 | 画面 | 目标信息 | 准备方式 |
| --- | --- | --- | --- |
| 1 | 主界面初始手牌 | 五大区域、8 张手牌、目标分数、上头值 | `http://localhost:8000/web/?seed=steam-shot-main` |
| 2 | 摊牌结算中 | 已出牌、筹码/倍率变化、分数上涨 | 选择 5 张牌后点击摊牌，抓数字动画中段 |
| 3 | 红眼赌注弹窗 | 三选一、红眼危险感、不是常驻 UI | 调试或真实流程把上头推入红眼后点击入口 |
| 4 | 暗涌翻牌 / 鬼压桌 | 红眼赌注的不确定风险 | 选择红眼赌注后摊牌，抓暗涌揭示瞬间 |
| 5 | 爆牌失败 | “分数够也可能死”的风险承诺 | 高上头值时摊牌触发爆牌 |
| 6 | 通关收益 | 一轮完成后的奖励反馈 | 达标后抓结算收益弹窗 |
| 7 | 商店阶段 | 赌资、赌鬼、降压道具和下一轮入口 | 通关后进入商店 |

首批 Steam 页面至少使用 5 张，其中 1、2、3、4、6 优先。爆牌截图可作为后续替换图，避免第一眼过度负面。

### 短视频 / GIF 脚本

目标长度：`15-30 秒`。

节奏：

| 时间 | 镜头 | 画面重点 |
| --- | --- | --- |
| 0-3 秒 | 手牌选择 | 玩家选中几张牌，左侧牌型/筹码/倍率预览变化 |
| 3-7 秒 | 摊牌结算 | 牌飞到中央，分数上涨，上头值同步上涨 |
| 7-11 秒 | 红眼进入 | 红眼入口解锁，上头值逼近危险区 |
| 11-16 秒 | 红眼赌注 | 三选一弹窗，选择 `借鬼钱` 或 `红眼翻倍` |
| 16-22 秒 | 暗涌翻牌 | 暗涌牌揭示，上头值跳动，玩家感到“要炸” |
| 22-28 秒 | 结果 | 二选一剪辑：极限通关收益，或爆牌失败 |

视频字幕建议：

```text
凑牌型，赚筹码。
分数越高，上头越危险。
红眼已开，要不要再押一手？
暗涌翻开。
过线，或者爆牌。
```

## 上线前手动 QA 清单

当浏览器流程测试无法作为唯一证据时，至少手动确认以下内容。每次公开试玩前记录 commit、入口 URL 和检查结果。

### 启动与布局

- 打开 `/web/` 后没有白屏或脚本报错。
- 画布保持 `1600 x 900` 等比缩放，没有页面滚动条。
- 左侧信息栏、顶部赌鬼区、中央出牌区、底部手牌区、右侧操作区都在预期位置。
- 手机横屏视口下手牌和右侧操作区没有越界；自动化布局用例覆盖 `932 x 430`，真机仍需人工复核可读性。
- 主要图片资源可见：桌面背景、普通牌背、红眼入口、上头眼睛、暗涌牌。

### 核心操作

- 初始发 8 张手牌，右侧牌堆数量正确。
- 点击手牌能选中/取消，最多选择 5 张。
- 选牌后左侧当前牌型、筹码、倍率会预览更新。
- 点击换牌后，选中牌进入弃牌堆，从牌堆补同等数量，换牌次数减少。
- 点击摊牌后，选中牌飞到中央出牌区并执行结算动画。
- 摊牌后当前分数、上头值、牌堆数量、手牌补牌状态正确更新。

### 红眼、失败与通关

- 上头值进入红眼区后，只解锁右侧红眼赌注入口，不自动弹窗。
- 玩家点击红眼入口后出现三选一弹窗，选择后只绑定下一次摊牌。
- 红眼赌注生效时会翻暗涌牌，并在本手结束后清除。
- 上头值达到爆牌阈值时显示爆牌失败，失败后主界面不可操作。
- 摊牌次数耗尽且分数不足时显示庄家通吃。
- 失败后点击再来一局能重置 demo。
- 分数达标且未爆牌时显示结算收益。
- 点击进入下一局后收益入账，进入商店或下一轮流程，红眼赌注本身不跨轮保留。

### 商店与发布

- 通关后商店可进入，购买/刷新/跳过不会卡住流程。
- 版本或构建 SHA 可在调试信息中定位。
- GitHub Pages 线上入口 `/web/` 能加载 `art/` 和 `public/` 下的资源。
