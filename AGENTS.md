# AGENTS.md

## 项目身份

本项目是一个纯 Web HTML/CSS/JS 原型，游戏名为《深渊赌局》，方向是黑暗扑克肉鸽。

当前 UI 方向固定为：

- 黑暗赌桌
- 扑克牌
- 赌鬼
- 上头值
- 红眼赌注
- 左侧信息栏
- 顶部赌鬼区
- 中央出牌区
- 底部手牌区
- 右侧操作区

除非用户明确要求，否则不要重做核心概念。

## 语言约定

后续所有文档、说明、注释、提交总结和面向用户的回复，默认使用中文。

除非用户明确要求英文或其他语言，否则不要再新增英文文档或英文说明。

保留代码标识符、文件名、命令、API 名称和必要的英文专有名词。

## 核心设计原则

这不是响应式网站。

这是固定游戏 UI 画布。

使用固定 16:9 画布：

- 基准尺寸：`1600 x 900`
- 整个画布根据视口等比缩放
- 不允许单个面板独立重排

## 硬规则

除非明确要求，否则不要修改以下内容：

1. 不移动左侧信息栏。
2. 不改变左侧信息栏的信息顺序。
3. 不在主游玩界面增加常驻商店 UI。
4. 不常驻显示红眼赌注选项。
5. 红眼赌注选项只能在触发时以中央弹窗出现。
6. 选择红眼赌注后，右侧只显示小图标，详情通过 tooltip 展示。
7. 不新增常驻面板。
8. 不重新引入拉杆/老虎机工业界面。
9. 不重命名核心术语：
   - 赌鬼
   - 上头值
   - 红眼
   - 鬼压桌
   - 红眼赌注
   - 摊牌
   - 换牌


## 视觉规则

UI 应该黑暗、可读、克制。

使用：

- 黑暗赌桌背景
- 旧纸牌
- 暗金边框
- 暗红高亮
- 红眼母题
- 只在选中或激活状态使用轻微发光

避免：

- 过量霓虹
- 赛博朋克 HUD
- 彩色噪点
- 常驻粒子效果
- 过亮的赌鬼卡
- 同一信息在多处重复展示

## UI 修改流程

当用户要求修改 UI：

1. 先确认要修改的是哪个现有组件。
2. 只修改该组件。
3. 保持无关布局、文字和视觉层级不变。
4. 不顺手“优化”无关区域。
5. 如果指令模糊，移动主要布局块之前先问。
6. 进入本次改动对应的真实游戏状态，在 `1600 x 900` 视口截图并人工检查；不能只依赖 DOM 断言或自动化测试通过。
7. 截图前等待关键图片、字体替代和动画状态加载完成，检查层级、裁切、重叠、亮度和文本可读性；发现异常先修复并重新截图。
8. 完成后说明改了什么、哪些受保护组件保持不变，以及截图检查了哪个状态。

## 实现规则

使用：

- 原生 HTML
- 原生 CSS
- 原生 JavaScript

优先使用 DOM 元素，方便检查和修改卡牌、按钮、tooltip、弹窗。

## 工具使用纪律

不要重复造轮子。截图、浏览器 QA、资源压缩、格式转换、发布检查等通用能力，优先使用已有成熟工具或项目内已经存在的测试流程。

只有当现成工具无法满足项目特有需求，并且用户确认后，才新增仓库内维护的辅助脚本。

## 验证清单

完成任何 UI 任务前，检查：

- 已进入改动对应的真实状态并完成截图检查
- 截图时关键图片已经加载，不把懒加载空白误判为最终效果
- 左侧信息顺序不变
- 没有重复展示分数/牌型信息
- 手牌区和出牌区视觉上区分清楚
- 红眼赌注未触发时隐藏选项
- 红眼赌注锁定入口的视觉权重低于“摊牌/换牌”
- 赌鬼卡不抢中央游玩区的视觉焦点
- 当前选中牌清晰可见
- 布局仍然完整适配 `1600 x 900`，没有滚动

## 文档更新纪律

除非明确要求，不新增文档文件。

规则、UI、视觉和领域文档集中维护在 `docs/` 下；根目录只保留项目入口、架构、任务和协作约束文档。

文档不是存档箱，而是后续任意对话理解全局的入口。维护职责：

- `PROJECT_CONTEXT.md`：全局入口、改动导航、运行和验证方式。
- `ARCHITECTURE.md`：实现分层、状态流、流程边界和测试边界。
- `docs/component_rules.md`：当前玩法和 UI 规则的主要事实来源。
- `docs/ui_spec.md`：主界面布局与受保护 UI 约束。
- `docs/design_tokens.md`：视觉令牌、本地资源和表现强度。
- `docs/decision_log.md`：设计原因，不维护最新功能清单。
- `docs/changelog.md`：历史变更记录，不作为最新规则来源。

新对话接手时，先读 `PROJECT_CONTEXT.md`，再读 `docs/component_rules.md`；只有涉及流程、状态或测试边界时再读 `ARCHITECTURE.md`。具体改动按 `PROJECT_CONTEXT.md` 的“改动导航”定位代码和同步文档。

代码改动后的固定流程：

1. 判断是否改变规则、流程、UI 约束、资源路径或测试方式。
2. 若改变，按 `PROJECT_CONTEXT.md` 的改动导航同步对应文档。
3. 若未改变，也在交付说明中写明文档无需更新。
4. 规则改动必须优先补或更新测试。
5. 面向用户总结时说明代码改了什么、文档改了什么、验证跑了什么。
