# 2026-05-30 开发记录与经验总结

## 今日真实状态

- Web MVP 已经从早期模块槽玩法收敛到扑克手牌玩法：52 张标准牌、8 张手牌、最多选 5 张出牌。
- 初始局面不再赠送 Joker；红温流 Joker 进入商店刷新池，由玩家购买后才生效。
- 牌型预览只显示牌型基础分和倍率，不提前展示单牌筹码、压力变化、Joker 增益或最终得分。
- 选中手牌时只在手牌区显示选中态，出牌区保持为空；只有点击出牌后，打出的牌才进入出牌区并逐张结算。
- 通过盲注后进入提现环节，再进入商店；提现包含盲注奖励、剩余出牌次数奖励和利息。
- 左侧 HUD 已按参考效果图重构为回合、比赛信息、目标分、当前得分、牌型读数、金币、压力、剩余出牌 / 弃牌。
- 商店、卡包、牌型升级、Boss 负面规则、抽牌堆 / 弃牌堆和 seed RNG 均已落地。

## 今天沉淀的设计经验

- UI 信息必须服务核心循环。主界面只放“能不能过、还剩几手、压力多危险、钱够不够买”的信息，其余内容放到比赛信息弹窗或结算日志。
- 预览和结算要分层。预览只给玩家做出牌判断，具体筹码变化、倍率变化、压力变化和 Joker 触发必须在出牌后逐步表现，否则会削弱结算爽感。
- 选牌区和出牌区不能混用。选牌是意图，出牌是承诺；如果选中手牌就复制到出牌区，会造成“两套牌同时存在”的违和感。
- 基础牌必须保持朴素。52 张标准牌已经足够支撑牌型判断，复杂度应该放在 Joker、Boss、商店和红温压力上。
- 红温 Joker 应该是构筑目标，不是初始能力。初始无 Joker 能让商店决策更重要，也避免玩家一开局就被规则文字压住。
- 参考效果图时先还原信息结构，再调材质。左侧 HUD 的顺序、权重和占屏比例比单个边框效果更重要。
- 每次 UI 大改后要用固定 seed 截图和交互验证。`?seed=preview-check` 用来固定手牌，点击选牌后检查出牌区是否为空，能快速避免回归。

## 当前验证口径

- 本地入口：`http://localhost:8000/web/?seed=preview-check`
- 逻辑检查：`node --check web/app.js`
- 纯逻辑测试：`node --test web/tests/logic-pure.test.mjs`
- 提交前检查：`git diff --check`

## 下一步建议

1. 用 3-5 个固定 seed 做平衡回放，记录是否能稳定出现“想卡红区”的行为。
2. 强化 80、90、95、100 压力节点的音效、震动、数字跳动和结算节奏。
3. 收敛 CSS 覆盖层，后续把主界面、商店、结算、弹窗拆成更清晰的样式段，降低继续复刻 UI 时的回归风险。
4. 为 shop、payout、selection 添加 DOM 级回归检查，避免“选牌又跑到出牌区”这类问题复发。
